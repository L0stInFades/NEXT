# 08 程序员开发流程：先做什么、接着做什么

## 8.1 这份文档解决的问题

面向“从 0 自研引擎 + 两宋首作”的落地现实，本章给程序员一个**可执行的开发顺序**与**工程流程**，避免：
- 一上来就做大系统（GI/生成/PT）导致主干不可运行；
- 先写玩法脚本、后补底层（Job/Asset/流式）导致全面返工；
- 系统之间没有边界，最后变成屎山。

核心规则：**主干永远可运行（可打包/可回归），研发功能必须可开关可回滚**。

## 8.2 工作方式（团队协作流程）

### A) 分支与合并（建议：Trunk-based）
- `main`：随时可运行、可回归（禁止长时间 broken）。
- 功能分支：短生命周期（1–3 天），PR 合并到 `main`。
- 合并前必须：
  - 通过构建
  - 通过最小回归（启动/跑一段固定轨迹）
  - 新增系统具备最小可观测性（至少日志/统计/调试面板之一）

### B) ADR/RFC（架构决策记录）
任何影响“模块边界/数据格式/脚本 API/渲染管线/世界分区”的改动，必须先写 1 页决策记录（可放在后续新增的 `docs/adr/`）：
- 选择了什么
- 为什么
- 备选方案
- 风险与回滚方式

### C) 双轨机制（Production vs R&D）
- Production：默认开启、决定交付。
- R&D：必须可开关（build flag + runtime flag），并提供 fallback。

## 8.3 “关键路径”开发顺序（Critical Path）

下面按依赖关系给出顺序；每一步都给出“交付物/验收口径”，确保你能判断“做完了没有”。

### CP0：工程骨架与最小可运行样例（先让项目能跑）
依赖：无

交付物：
- 清晰的目录分层（Platform/Foundation/Runtime/Feature/Tools/Game）
- 一键构建 + 一键运行的“空场景”程序（窗口 + 相机移动）
- 崩溃捕获与日志输出（至少能定位 callstack）

验收口径：
- 新人拉仓库 30 分钟内可编译运行；
- 运行 5 分钟不崩溃，日志可用。

### CP1：可观测性优先（Profiler/统计/调试面板）
依赖：CP0

交付物：
- CPU 采样/计时（Frame、系统级 scope）
- GPU 时间戳框架（哪怕只显示总 GPU 时间）
- 内存/VRAM/IO 的基础统计接口（先不求完美，要求能扩展）

验收口径：
- 能回答三个问题：这帧 CPU 花在哪？GPU 花在哪？内存/IO 有没有异常尖峰？

### CP2：Job System（并行化与预算化的地基）
依赖：CP1（至少要能测任务耗时）

交付物：
- 任务队列、优先级、依赖/栅栏
- 可观测：队列长度/耗时分布/取消与超时（至少统计）

验收口径：
- 能把一个“模拟任务”（比如生成一堆假工作）稳定分摊到多核且不拖垮主线程。

### CP3：Asset Pipeline 雏形（离线编译 + 运行时加载）
依赖：CP2（编译/解压/IO 都要任务化）

交付物：
- `assetc`（资源编译器）能把少量资源编译成运行时格式（先只做：mesh/texture/material 的最小集）
- 运行时 Asset Runtime：异步加载、引用计数/句柄、卸载
- 打包容器雏形：支持顺序读、带索引

验收口径：
- 运行时不再直接读 FBX/PNG；
- 同一资源加载多次不重复 IO；卸载不崩溃。

### CP4：Runtime World（ECS/Transform/事件总线）
依赖：CP2/CP3（实体创建与资源句柄需要异步能力）

交付物：
- ECS 最小集：Entity、Component、Query、System 调度
- Transform（父子层级/挂点）
- Event Bus（发布/订阅）+ 事件记录开关（为回放铺路）

验收口径：
- 能在世界里动态创建/销毁实体；
- 跨系统通过事件通信而不是互相硬引用。

### CP5：渲染基线（先 Raster 跑通，再谈高级）
依赖：CP3/CP4/CP1

交付物：
- RHI（DX12）最小可用
- Render Graph 骨架（pass/依赖/资源生命周期）
- 基础 PBR + 阴影 + 后处理 + 调试视图

验收口径：
- 能稳定渲染一个小场景并展示 GPU 分解耗时；
- 支持 Golden Image 对比（先手动也行）。

### CP6：手感与运镜（尽早锁定差异化）
依赖：CP4/CP5

交付物：
- Camera Graph（第一/第三/锁定/轨道）可混合切换
- 输入系统与控制器参数化（响应曲线、阻尼、惯性）

验收口径：
- 设计师能调参数并保存预设；
- 10 分钟跑跳切换镜头不抖、不穿帮。

### CP7：世界分区与流式（首作最大风险点，越早越好）
依赖：CP2/CP3/CP4/CP5（缺一不可）

交付物：
- World Partition（Grid Cell）+ Cell 清单/索引
- 分层加载：先形体/HLOD，后细节纹理/装饰
- 预算面板：IO/解压/GPU 上传/实体实例化

验收口径：
- 固定相机轨迹跑 5–10 分钟无硬读条；
- 任意卡顿可定位到具体 Cell/资源。

### CP8：脚本系统（先跑通“最小闭环”）
依赖：CP4（事件与世界 API）、CP3（脚本资源加载）

交付物：
- Lua（或选定方案）嵌入、热重载（开发期）
- 白名单 API（世界查询/生成/触发事件）
- 脚本性能预算统计（至少每帧耗时）

验收口径：
- 能用脚本在运行时生成/控制实体，并能被卸载/回滚。

### CP9：任务系统（声明式条件 + 灾害扰动用例）
依赖：CP4/CP8/CP7（世界状态可查询，才能“不断支线”）

交付物：
- 任务数据模型（目标/约束/执行步骤可重规划）
- 至少一个“灾害扰动后改道”的演示任务
- 任务调试 UI：当前目标、卡住原因、被哪些事件影响

验收口径：
- 同一任务在不同世界状态下都能闭环，不出现硬掐支线。

### CP10：编辑器与生产化（让内容吞吐起来）
依赖：CP3/CP4/CP5/CP7/CP9

交付物：
- 编辑器基础：场景/分区可视化、属性面板、调试面板挂载
- 资源验证器：预算检查、规则检查（先少量规则）

验收口径：
- 美术/设计能独立做一个小区域并跑通任务闭环。

## 8.4 可并行的工作拆分（避免互相阻塞）

当 CP0–CP2 完成后，就可以开始并行：
- 渲染组：CP5（RHI/Render Graph/基础材质）
- 世界组：CP7（分区数据结构/预取策略/预算面板）
- Gameplay 组：CP6（镜头/手感/控制器）
- 工具组：CP10 的编辑器外壳、导入/验证器
- 脚本/任务组：CP8/CP9（在“假世界/假事件”下先跑通流程，再接入真实世界）

并行的前提：所有组都必须走**稳定接口**（见 `docs/01-engine-structure.md` 的模块契约思想）。

## 8.5 每个系统的 Definition of Done（DoD）

任何系统进入 `main`，至少满足：
- 有最小示例场景/用例（能运行并展示效果）
- 有统计/日志/调试入口（可观测）
- 有降级/关闭开关（尤其是实验性功能）
- 有性能预算说明（哪怕是“当前未知，先占用 X ms/MB，后续压缩”也要写出来）
- 不破坏固定轨迹回归（启动 + 跑一段轨迹）

## 8.6 什么时候开始做“高级功能”（避免顺序错）

建议的触发条件（满足再做，否则先别做）：
- 混合 RT / PT：至少在 CP5/CP7 稳定后再做；否则你会在缺少流式与预算体系时被性能问题拖死。
- AI 生成（运行时）：先把离线生成/缓存/验证器做起来，再考虑运行时；否则内容不可控。
- 百年时间线：先把 Event + Snapshot 的基础能力做出来（CP4/CP7/CP9），再扩展 Epoch 与聚合系统。
