# 开发交接：CP2 → CP3

## 1. 当前状态（CP2 已完成）
- Job System：多线程任务队列、High/Normal/Low 优先级、依赖入队、主线程预算泵 `Pump(budgetMs)`、统计接口 `GetStats()`。
- 集成：游戏启动自测 32 任务；主循环每帧 0.25ms 泵任务。
- 仍缺：取消/超时、任务命名/分组统计、无锁队列优化、Profiler 线程安全。

## 2. CP3 目标回顾（Asset Pipeline 雏形）
- 资源编译器 `assetc`：将少量 mesh/texture/material 转换为运行时格式。
- 运行时 Asset Runtime：异步加载、引用计数/句柄、卸载。
- 打包容器：带索引的顺序读容器，支撑流式。
- 全程任务化：编译/解压/IO 均使用 Job System。

## 3. 建议的首个迭代（1–2 天）
1) 资产格式约定（最小集）  
   - 定义 mesh/texture/material 的中间格式（头信息 + 数据块 + 版本号）。  
   - 写简易文档和验收样例（1–2 个测试资产）。
2) Asset Runtime 骨架  
   - 句柄类型、引用计数、加载/卸载 API（异步返回 Future/Handle）。  
   - Job System 集成：IO 解压任务、GPU 上传任务预留接口（GPU 部分可空实现）。
3) 简易打包容器  
   - 目录 + 数据区，支持顺序读 + 偏移索引；支持从文件或内存映射读取。  
   - 校验：版本、校验和。
4) 验收样例  
   - 在 demo 中加载一个打包文件，模拟读取两份资源并卸载，打印句柄/引用计数/耗时日志。

## 4. 先修技术债（优先级高）
- Job System：补充取消/超时、任务命名（便于统计/调试）；考虑无锁队列或减少锁粒度。  
- Profiler：添加基础锁或线程本地缓冲，避免多线程数据竞争。  
- logger/平台：日志落盘以便并行任务的崩溃回溯。

## 5. 辅助原则
- 所有 IO/编译任务走 Job System；主线程仅做调度与验证。  
- 资源句柄全程不可泄漏，卸载路径需有日志与断言。  
- 每个新格式都带版本号与最小迁移脚本占位。  
- 验收以可运行样例为准：构建后能加载/卸载并打印统计。

## 6. 验收口径（CP3）
- 运行时不直接读 FBX/PNG；通过编译产物加载。  
- 同一资源多次加载不重复 IO，卸载不崩。  
- 任务化 IO 不阻塞主线程，Job System 统计可见。
