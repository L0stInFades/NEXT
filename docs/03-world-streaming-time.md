# 03 超大世界：分区/压缩/无加载与百年时间线

## 3.1 目标

- 超大地图：支持城市群、乡野、河道、山地与大规模战场。
- **无加载体验**：尽量消灭“硬切读条”，允许在镜头与剧情上实现一镜到底。
- **高压缩 + 高吞吐**：资源体积可控、加载稳定、可预测。
- **百年时间线**：世界状态能表达长跨度变化（建筑兴废、植被变化、灾害影响、人口迁徙等），并与任务系统联动且不脆断。

## 3.2 世界分区（World Partition）

推荐使用“分层网格 + 内容层（Layer）”：
- **Grid Cell**：最小流式单位（大小由性能与密度决定）。
- **Region/Chunk**：更高一层，用于预取与 HLOD 生成。
- **Layer**：按功能划分（地形、建筑、道具、NPC 出生点、任务点、导航、HLOD、音频等）。

每个 Cell 的数据建议拆成：
- **Static Pack**：地形/静态网格/装饰（多为只读，可强压缩）。
- **Dynamic Seed**：动态生成的种子与规则（可决定 NPC/植被/小物的分布）。
- **Runtime State Delta**：运行时变化的差量（玩家影响、灾害影响、脚本事件）。

## 3.3 资源虚拟化：把“世界”当作资源流

无加载的大前提：世界不仅是“实体”，更是“资源句柄的集合”。

需要统一的流式管线：
1. 兴趣点（Interest）计算：相机 + 预测路径 + 任务/剧情需求。
2. 生成加载请求：Cell → Asset bundles → 子资源（纹理页/mesh cluster/动画片段）。
3. IO 读取：顺序读优先；支持批量请求合并。
4. 解压与反序列化：异步任务化，带预算。
5. GPU 上传：copy queue；限速；避免抢占渲染。
6. 实体实例化：把资源句柄挂到 ECS 上；可逐步细化（先粗后细）。
7. 淘汰（Eviction）：超预算按优先级卸载；必须无崩溃、无悬挂引用。

## 3.4 预测与预取（为一镜到底服务）

相机不仅决定当前可见，还决定“未来几秒要看到什么”：
- 输入：相机速度/方向、镜头脚本轨道（cutscene）、玩家意图（例如骑马/快跑）、任务目标。
- 输出：未来 N 秒的兴趣区域体积（frustum sweep / corridor）。

预取策略必须可配置：
- “镜头轨道”权重最高（剧情一镜到底必须保障）。
- “玩家自由移动”用速度与地形约束预测。
- 当预算不足时：先加载低 LOD/HLOD，保证连续性，再补高细节。

## 3.5 压缩与打包（Asset Packaging）

目标：**减少随机 IO、减少 CPU 解压尖峰、支持增量更新与 Mod**。

建议：
- 打包单位：按 Region/Cell 的 bundle（同时包含 mesh/texture page/material/nav 等索引）。
- 压缩算法：
  - Production：zstd（易用）或商用方案（若可行）用于更优吞吐。
  - 纹理：优先 GPU 原生压缩（BCn），减少运行时 CPU 解压。
  - 几何：可选 Draco/自定义量化压缩（需要评估解码成本与 GPU 友好性）。
- “字典压缩”：对历史题材的大量相似资产（瓦片/木构/砖石纹理）可能收益明显。

验收口径：
- 同一段相机轨迹多次运行，加载带宽与卡顿必须高度稳定（可重复性）。

## 3.6 LOD/HLOD：规模感与性能的关键

### A) 远景策略
- HLOD：对建筑群/树林/城墙等进行层级合并与材质烘焙（或虚拟纹理）。
- Impostor：对极远景/植被/人群使用 billboard/简化体积。
- 大气与雾：用体积与大气散射“吃掉”远景细节，反过来降低几何压力（与美术协同）。

### B) 近景策略
- 近景高细节依赖流式补齐：先出现“完整形体”，再加载高频纹理/细节网格。
- 碰撞与交互：可与渲染 LOD 解耦（交互优先保障）。

## 3.7 百年时间线：世界状态的表达方式

“百年变化”不能靠每秒模拟 100 年，那会在存档、性能、任务系统上全面爆炸。

推荐：**事件溯源（Event Sourcing）+ 分层世界状态 + Epoch（时代段）**。

### A) 三层世界表示
1. **基准层（Baseline World）**：一个参考年份的“静态世界”（地形/主要城镇格局/重要建筑）。
2. **历史层（Historical Events）**：跨时间段的“变更事件”（建造、毁坏、迁徙、政令、灾害）。
3. **玩家层（Player Delta）**：玩家导致的差量（任务选择、破坏/修复、关系变化）。

### B) Epoch（时代段）与 LOD
把 100 年切成若干 Epoch（例如 5 年/10 年一段，视需求）：
- 每个 Epoch 可以有：
  - 世界差量快照（用于快速跳转/加载）
  - 关键事件索引（供任务系统查询与回放）
  - 低频系统的“聚合结果”（例如人口、繁荣度、季节性资源）

### C) “可回放”与存档策略
存档建议：
- 定期世界快照（Snapshot）
- 快照之后的事件流（Event Log）
优势：
- 可复现复杂 bug（尤其是灾害扰动与任务互相影响时）
- Mod/补丁更容易做兼容迁移（有版本与事件）

## 3.8 大场面（人群/军阵）与世界系统的协同

大场面必须做“模拟 LOD”：
- 近景：完整 AI/动画/物理（数量受预算限制）
- 中景：简化 AI（队形/集群），动画实例化，碰撞简化
- 远景：纯渲染群体（GPU instancing + 预烘焙动画/运动向量），只保留宏观态势

战场与城市同理：远处不要“真 NPC”，要“群体表现”。

## 3.9 调试工具（世界系统必须可视化）

必备：
- 分区网格可视化（加载中/已加载/即将卸载）
- 流式队列与预算面板（IO/CPU 解压/GPU 上传）
- 热点分析：哪个 Cell/资产导致尖峰
- 相机轨迹回放：用于复现“某镜头必卡”的问题
