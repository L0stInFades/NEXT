# 01 引擎分层、模块边界与运行时模型

## 1.1 总览：分层与依赖方向

核心要求：**依赖单向、层次清晰、可替换点明确**，避免“任何改动都牵一发而动全身”。

推荐分层（从底到顶）：
1. **Platform**：OS/窗口/输入/文件系统/线程/时间/动态库加载/崩溃捕获。
2. **Foundation**：内存/容器/数学/字符串/日志/配置/Profiler/Job System/反射与序列化基础。
3. **Runtime Core**：ECS/场景与 Transform/资源系统（Asset Runtime）/事件总线/调度（tick）。
4. **Feature Modules**：Renderer、World Streaming、Physics、Animation、AI、Audio、Networking（可选）。
5. **Gameplay Framework**：任务/脚本/交互/战斗/叙事/世界规则（偏“游戏通用层”）。
6. **Tools**：编辑器、导入器、资源编译器、调试与回放工具、性能回归工具。
7. **Game Project**：两宋首作的具体内容与玩法实现。

强约束：
- Engine Core 永远不依赖 Game。
- Tools 可以依赖 Engine（并复用 Runtime），但 Runtime 不依赖 Tools（避免“编辑器逻辑进运行时”）。
- Feature Modules 之间依赖要尽量通过接口与数据边界；禁止互相 include 一堆私有实现。

## 1.3 运行时的“最小共同语言”（四大件）

为了支撑超复杂世界、无加载、百年时间线与 Mod，运行时必须尽早统一以下四个基础能力：

### A) Job System（任务系统）
- 目标：把“可能卡主线程”的事情都变成可预算、可并行、可取消的任务。
- 需要具备：工作窃取、优先级、依赖/栅栏、纤程/协程（可选）、可观测（队列长度、耗时分布）。

### B) Asset Runtime（资源运行时）
- 目标：所有资源都可“虚拟化 + 流式 + 版本化”。
- 资源类型：几何、纹理、材质、动画、音频、脚本字节码、导航块、世界分区数据、HLOD 包、AI 生成缓存等。
- 关键点：
  - **离线编译**：运行时不做重解析（例如不直接读 FBX/PNG）。
  - **可热更新**：开发期支持热重载；发行版可用于 Mod 加载。
  - **版本迁移**：格式 schema/versioning 与迁移工具是长期生命线。

### C) ECS + World State（实体组件与世界状态）
推荐：**ECS 作为权威模拟层**，并在其上建立“可查询、可回放”的世界状态。

为什么不只用传统 Scene Graph：
- 超大世界与海量 NPC/物件时，数据局部性与并行更新更关键。
- 任务/灾害扰动/百年时间线需要“世界状态可查询、可订阅、可回放”。

建议的折中：
- ECS 负责：逻辑、AI、任务、交互、物理、动画驱动参数。
- Scene/Transform 层：作为 ECS 的一组组件与系统（支持父子层级、骨骼、挂点）。
- Rendering Scene：渲染侧维护独立的“可 GPU 驱动”的视图数据（来自 ECS 的同步/提取）。

### D) Event Bus（事件总线）
- 目标：让“世界发生了什么”成为第一等公民，而不是散落在 if/else 里的副作用。
- 用途：
  - 任务系统：订阅事件、派生目标、做韧性重规划。
  - 世界模拟：灾害/政令/战争/经济波动等以事件形式表达。
  - 调试与回放：记录事件流 + 定期快照即可复现复杂 bug。

## 1.4 Tick 模型：固定步模拟 + 渲染解耦

推荐：
- **Simulation Tick（固定步）**：AI/物理/任务/规则/战斗等尽量固定步，保证稳定性与可回放性。
- **Presentation Tick（可变步）**：镜头、动画插值、VFX、UI 等可变步，提升观感与输入响应。
- **Render Frame**：与 Presentation 解耦，通过 Render Graph 执行。

关键点：
- 输入的采样与响应要“前置”，镜头/角色控制的“手感”由输入响应曲线与阻尼系统统一管理。
- 流式加载要在 Tick 中有固定预算（CPU/IO/GPU 上传），并允许被相机预测驱动（见世界流式文档）。

## 1.5 模块边界：用接口与数据墙隔离复杂度

每个大模块都要明确三件事：**输入数据、输出数据、生命周期**。

建议的“模块契约”（概念层，不是具体代码）：
- `IRenderer`：接收一帧的 Render Scene（相机、灯光、可见物、材质参数、天气等），输出图像/调试纹理/统计。
- `IWorldStreamer`：接收“兴趣点/相机轨迹预测/预算”，输出“已加载的世界分区实体与资源句柄”。
- `IPhysics`：接收 ECS 的物理组件与约束，输出刚体状态/触发事件。
- `IAnimation`：接收骨骼/状态机/IK 请求，输出姿态（供渲染与物理/命中使用）。
- `IAI`：接收感知输入与世界状态查询接口，输出行为决策与行动请求（移动、交互、对话、战斗）。
- `IScripting`：接收世界 API 与事件，输出脚本驱动的行为（任务推进、生成、规则改写）。

原则：
- 模块之间通过“瘦接口 + 数据句柄”沟通，不传裸指针到处跑。
- 所有跨模块的长生命周期对象必须可追踪（ID/Handle），支持卸载与重载。

## 1.6 反射、序列化与版本化（编辑器/Mod 的地基）

如果要做高质量 Mod 与长期项目，必须把“数据是第一公民”落实到：
- **反射**：属性可枚举、可编辑、可序列化、可复制、可网络同步（可选）。
- **序列化**：二进制为主（快），文本为辅（调试与 Mod 编辑）。
- **版本化**：任何资源/存档/脚本接口都要带版本；提供迁移路径（哪怕最初是简单脚本迁移）。

推荐策略：
- 引擎内部用二进制格式（自定义或 FlatBuffers/Protobuf 一类），并提供工具把它导出成可读的“Mod 源格式”。
- 存档采用：**快照 + 事件流**（事件溯源），为“百年时间线/灾害扰动/任务韧性”服务。

## 1.7 可观测性（不要等做完再补）

最低配观测能力（上线前必须）：
- CPU/GPU Frame Profiler（含 Render Pass 粒度）
- 任务系统统计（队列、耗时、依赖、取消）
- 内存/VRAM 统计（分配器级、资源级）
- IO 统计（带宽、延迟、缓存命中、解压耗时）
- 世界流式可视化（分区格、加载状态、预算消耗）
- 事件流/任务系统的录制与回放（开发期）

## 1.8 Production Track vs R&D Track：工程机制

为了允许“先进渲染/新压缩/生成式 AI/新材质”等不断试验而不拖垮主干：
- 所有实验性功能必须：
  - 有明确的开关（build flag + runtime flag）
  - 有回退路径（fallback）
  - 有指标（画质收益/性能代价/内存代价/内容制作代价）
- 任何进入 Production 的功能必须：
  - 通过性能基准与稳定性基准
  - 有工具链支持（至少 debug/统计）
  - 有资产规范与验证器（避免“内容把引擎喂死”）
