# 05 逻辑脚本与任务系统：复杂世界、随机灾害与“不断支线”

## 5.1 目标

- 让任务系统适配“超复杂世界 + 随机灾害扰动 + 百年时间线”，避免传统关卡式脚本的脆弱性。
- 支持：
  - 任务可重规划（灾害/战乱/人物死亡/建筑毁坏时不直接掐断）
  - 世界事件对任务产生可解释影响（不是暗箱判定）
  - Mod 作者能做高质量任务，并与主线系统联动

## 5.2 核心理念：事件驱动 + 声明式条件 + 可回放

### A) 事件驱动（Event-first）
- 世界发生的变化都以事件形式进入总线（Event Bus）。
- 任务只订阅事件与查询世界状态，不直接持有脆弱引用（例如某个关卡对象指针）。

### B) 声明式条件（Declarative Conditions）
任务目标建议写成“条件集合”：
- 条件：时间窗口、地点、人物关系、物品状态、城镇状态（繁荣/戒严/灾后等）
- 动作：触发对话、生成 NPC、改变世界规则、派发奖励

优势：
- 世界变化后，系统可以解释“为什么目标变化/延后/替换”。

### C) 可回放（Replay）
开发期必须可回放：
- 事件流 + 关键快照
用于定位：灾害/随机性引发的“只在某次出现”的任务 bug。

## 5.3 脚本系统选型建议（Production Track）

在“可嵌入、可热更、Mod 友好、性能可控”之间取平衡：

推荐基线：**Lua（或 LuaJIT）+ 强约束 API + 数据驱动图编辑器**
- Lua：嵌入成熟、热更方便、Mod 生态好。
- 强约束 API：通过绑定层暴露“白名单能力”，避免脚本随意触底层导致稳定性问题。
- 类型与校验：建议通过代码生成（schema → bindings）提供静态检查/运行时校验。

可选路线：
- C#（需要更重的运行时与部署考量，但工具链友好）
- 自研 DSL（成本高，但可针对任务/规则做极强约束；适合后期）

结论建议：
- 首作先用 Lua + 图编辑器落地 Production；
- R&D 再探索 DSL/更强的规划器。

## 5.4 任务系统的数据模型（避免“夜曲式截断”）

把“任务”拆成三层：
1. **叙事层（Narrative Intent）**：讲什么故事、要达成什么体验（不直接绑定实现路径）。
2. **目标层（Goals & Constraints）**：以声明式方式描述达成条件与限制（时间/地点/人物可用性）。
3. **执行层（Plans/Steps）**：当前世界状态下的具体步骤（可替换、可重规划）。

当世界扰动发生（灾害/人物离开/地点封锁）：
- 目标层不变或小变；
- 执行层允许重新规划：
  - 换联系人
  - 换地点
  - 延后到合适时间窗口
  - 转为“补救任务”（灾后重建/寻找失踪者）

这套机制要求：
- 世界状态可查询（见世界文档的事件与快照）
- NPC/地点可用性可被系统评估（而不是脚本写死）

## 5.5 随机灾害与世界规则系统

建议把“灾害/政令/战争”等做成独立的 **World Rule Modules**：
- 输入：时间、地理、社会状态、玩家行为（例如赈灾影响）
- 输出：事件（洪水、瘟疫、粮价波动、治安下降等）+ 世界状态差量

任务系统处理方式：
- 订阅灾害事件
- 用规则把任务目标映射到“新条件/新步骤”

关键点：
- 灾害不能把任务系统“写死”；必须通过统一事件与规则接口影响世界。

## 5.6 设计师工作流（工具链要求）

必须提供：
- 任务图编辑器（节点：条件、动作、分支、计时器、监听事件）
- 任务模拟器：在不同世界状态/不同灾害下跑通（自动化验证）
- 日志与可视化：任务当前目标、为什么卡住、被哪些事件改变

## 5.7 Mod 支持的脚本边界

Mod 脚本必须：
- 运行在沙盒能力集里（白名单 API）
- 有性能预算（每帧脚本耗时、事件处理上限）
- 受版本化接口保护（API 变更可迁移）

建议提供：
- 官方“任务模板”与验证器（避免低质量 Mod 拉垮体验）
- 资源打包与依赖管理（Mod Manifest）
