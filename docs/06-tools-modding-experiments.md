# 06 工具链、Mod SDK 与可持续实验体系

## 6.1 工具链的定位

首作在两宋，内容规模与复杂度会很高；工具链不是“锦上添花”，而是决定能否交付的主生产力。

工具链的三大目标：
- **高吞吐**：资产导入/编译/验证/打包要快，支持增量与并行。
- **高确定性**：同输入产出同结果（便于回归、CI、Mod 兼容）。
- **高可观测**：任何卡顿、加载尖峰、脚本异常、流式抖动都能定位。

## 6.2 编辑器（Editor）

建议路线：**自研轻量编辑器 + 运行时复用**（避免“编辑器与运行时两套世界”）。

### A) 编辑器必须覆盖的能力（按优先级）
- World 编辑：分区网格、地形/水系、建筑摆放、HLOD 预览
- Entity/Component 检视与编辑：反射驱动 UI（属性面板）
- 场景流式预览：相机轨迹回放、预取/预算可视化
- 渲染调试：Render Graph、材质/纹理页、光照探针可视化
- AI 调试：黑板/决策/行为树可视化、群体 LOD 状态
- 任务编辑：任务图（节点/条件/动作）、模拟与回放

### B) 技术建议
- UI：ImGui 起步（快），后期可替换更重 UI 框架，但不要让 UI 侵入引擎核心。
- 编辑器功能“插件化”：每个大系统提供自己的调试与编辑面板。

## 6.3 资源管线（Asset Pipeline）

核心原则：**离线编译、运行时只加载“引擎格式”**。

### A) 建议的管线阶段
1. Import：从 DCC（FBX/GLTF/EXR/PNG/WAV 等）导入到中间格式（可带元数据）。
2. Cook/Compile：编译成运行时格式（mesh clusters、纹理页、材质实例、导航块等）。
3. Validate：自动检查预算与规范（poly/骨骼/贴图格式/命名/依赖）。
4. Package：按 Cell/Region 打包（服务世界流式与 Mod）。

### B) 增量与缓存
- 每个资源产物必须有内容哈希与依赖哈希（deterministic build）。
- 支持“只编译受影响部分”（大地图内容必需）。
- CI 可跑“从零编译”与“增量编译”两种路径，防止缓存把问题掩盖。

## 6.4 Mod SDK：高质量 Mod 的工程底座

目标：允许 Mod 作者做到：
- 新任务/新 NPC/新建筑/新地图分区
- 与主线脚本系统联动（受控地影响世界）
- 资产与脚本能被验证、被预算限制、被版本迁移

### A) Mod 的构成（建议）
- `manifest`：Mod ID、版本、依赖、兼容的引擎/游戏版本、权限声明（需要哪些 API 能力）。
- `content`：资源包（打包格式与官方一致，便于流式）。
- `scripts`：脚本/任务图/规则模块（运行在沙盒）。
- `localization`：文本与配音引用（可选）。

### B) 兼容性与版本化
- 引擎 API 与数据 schema 必须版本化；提供迁移工具（哪怕是“自动 + 需要人工确认”的半自动）。
- 资源格式版本变更要能兼容老 Mod（至少给出明确报错与升级路径）。

### C) 安全与稳定性（不可妥协）
- 脚本沙盒：白名单 API；禁止任意文件/网络/进程访问（除非明确允许）。
- 性能预算：每帧脚本耗时上限、事件处理上限、生成任务上限。
- 资源预算：VRAM/内存/实例数/粒子数上限与验证器。

### D) 高质量 Mod 的“官方护栏”
- 提供官方模板：任务模板、对话模板、建筑套件模板。
- 提供验证器：一键检查“时代约束/内容预算/依赖完整性”。
- 提供调试工具：任务回放、事件流查看、世界状态查询。

## 6.5 可持续实验体系（R&D 不拖垮 Production）

把“试验新技术”变成工程流程，而不是个人英雄主义：

### A) 插件化与 Feature Flags
- 每个实验模块必须：
  - 独立模块/插件（可单独编译/加载）
  - build flag + runtime flag（可在不重编译时关闭）
  - fallback 路径（关闭后仍能正常跑）

### B) 实验验收的三张表（进入 Production 前必须）
1. 画质收益：对比镜头/指标（截图序列 + 量化指标可选）
2. 性能成本：ms/带宽/内存/VRAM 增量
3. 制作成本：内容要怎么做、是否增加美术负担、是否增加调试难度

### C) 回归体系（防“悄悄变坏”）
- Golden Image（渲染）：固定相机轨迹输出序列帧做对比
- 性能回归：固定轨迹 + 固定配置，记录 CPU/GPU/IO/内存
- 稳定性回归：长时间运行（无加载、时间推进、灾害事件）观察泄漏与抖动

### D) 架构决策沉淀（建议建立 ADR/RFC）
- 任何“会影响模块边界/数据格式/脚本 API/渲染管线”的决策，都要留下简短记录：
  - 为什么这么选
  - 替代方案是什么
  - 风险与回滚路径
